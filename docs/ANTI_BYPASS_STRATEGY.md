# Стратегия защиты от обхода платформы

> Обход = сделка состоялась благодаря платформе, но деньги прошли мимо неё.
> Задача — не «запретить», а сделать обход **невыгодным и рискованным**,
> сохранив удобство для честных мастеров.

## 1. Векторы обхода и защита

| # | Вектор | Статус защиты |
|---|--------|---------------|
| 1 | Обмен контактами в чате → уход в наличку | детектор есть, **маскирование номеров ВЫКЛЮЧЕНО** ⚠️ |
| 2 | Занижение цены в приложении + доплата наличными | ✅ модерация всех снижений (новое) |
| 3 | Расчёт «по факту» с занижением по сговору | ✅ модерация + фрод-сигнал (новое) |
| 4 | Отмена заказа после получения контакта | ✅ детектор `checkCancelAfterContactPattern` |
| 5 | Первый заказ через платформу, дальше — напрямую | ✅ надбавка `first_order_commission_rate` |
| 6 | Самосделки (один телефон у клиента и мастера) | ✅ детектор `detectSelfDealing` |
| 7 | Оплата остатка наличными | ✅ не проблема: комиссия уже в депозите |

## 2. Главный экономический якорь (уже работает)

```
депозит = max(deposit_rate% × total, commissionAmount)
```

Депозит **всегда ≥ комиссии платформы** и списывается при создании заказа.
Даже если стороны потом договорятся о наличных — платформа **уже получила своё**.
Это сильнее любых запретов.

Дополнительно: комиссия берётся со **всех** расчётов, включая выезд
(`visit_fee_commission_rate = 15%`).

## 3. Растущая шкала комиссии — анти-обходная по замыслу

```
< 100k  → 10%     300k–800k → 14%
100–300k → 12%    ≥ 800k    → 15%
```

Мелкие заказы легко увести в наличку → комиссия низкая, обходить невыгодно.
На крупных ценность эскроу-защиты высока для обеих сторон → комиссия выше.

## 4. Что добавлено в модуле изменения цены

- **Любое снижение цены** → `MODERATION` + фрод-сигнал `PRICE_DECREASE_REQUEST`.
- **Расчёт по факту** → `MODERATION` + сигнал `SETTLEMENT_UNDERDECLARE`
  (подтверждения клиента недостаточно — при сговоре оба «за»).
- Рост > 20% за шаг → модерация; суммарный рост > 50% → запрещён.
- Обязательны причина и фото — модератору есть что проверять.
- Все решения пишутся в аудит и в fraud-скор мастера.

## 5. Приоритетные шаги (рекомендация)

### 5.1 Включить маскирование номеров — САМОЕ ВАЖНОЕ
Сейчас `virtual_numbers_enabled = false`. Пока номера открыты, вектор №1
(самый массовый) не закрыт: стороны просто созваниваются напрямую.

```
virtual_numbers_enabled = true
```

### 5.2 Привязать гарантию к платформе
Гарантия (`Guarantee`) действует **только** по заказам, проведённым в приложении.
Это главный аргумент для клиента не уходить в наличку: «мимо платформы —
никакой защиты, спора и возврата». Показывать это прямо в UI при оплате.

### 5.3 Сделать штраф применимым
`bypass_penalty_multiplier = 3` существует, но нужен механизм фиксации факта:
жалоба второй стороны + доказательства (переписка, фото) → админ применяет штраф.
Без процедуры доказывания множитель — «мёртвая» настройка.

### 5.4 Порог fraud-скора → авто-действия
`scanUser` даёт скор 0..100 (`clean` / `suspicious` / `high` / `critical`).
Предлагаю:
- `high` (≥60) → все заявки мастера на снижение цены только через модерацию (уже так) + флаг в админке;
- `critical` (≥80) → временная блокировка выдачи новых заказов до разбора.

### 5.5 Мотивация «оставаться в системе»
Запреты работают хуже пряника. Что усиливает удержание мастеров:
- PRO-подписка с **0% комиссии** (уже есть) — мастеру выгоднее работать вбелую;
- рейтинг/портфолио/поток заказов, которые теряются при уходе;
- быстрые выплаты.

## 6. Метрики для контроля

Отслеживать еженедельно:
- доля заказов с отменой после первого контакта (вектор №1);
- доля заявок на снижение цены и % отклонённых модератором;
- доля повторных пар «клиент↔мастер» вне платформы (косвенно: клиент перестал заказывать);
- средний fraud-скор активных мастеров.

## 7. Что НЕ рекомендую
- Жёсткие блокировки по одному сигналу — ложные срабатывания выжигают честных мастеров.
- Полный запрет снижения цены — бывают честные ситуации (работ оказалось меньше).
  Правильно — модерация, а не запрет.
