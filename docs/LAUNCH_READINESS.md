## 🔴 БЛОКЕРЫ — без них платформа не работает

### 1. Нет мастеров (247 клиентов на 4 мастера)

Маркетплейс не может работать при таком перекосе. Заказ публикуется — откликаться некому.
Из 6 профилей мастеров только **3 имеют категории**, а рассылка новых заказов идёт
по категориям → фактически заказ видят 3 человека.

**Решение:**
- Ручной набор 30–50 мастеров по ключевым категориям (сантехника, электрика, мебель,
  двери) до любого маркетинга на клиентов.
- Обязательная простановка категорий при регистрации мастера — сейчас можно
  зарегистрироваться без них и не получать заказы вообще.
- Регистрационный взнос 400 000 сум на старте — барьер. Сейчас он и так фактически
  не работает как гейт (мастер без оплаты не блокируется), но в интерфейсе отпугивает.
  Решить: либо отключить на период набора, либо убрать из онбординга.

### 2. Нет способа принять реальные деньги

- Payme: **сертификация в песочнице не пройдена**, боевых кредов нет.
  `PAYME_USE_SANDBOX` в проде не задана → код работает в боевом режиме
  с непроверенными реквизитами.
- Click: ключи заданы, но webhook не протестирован ни разу.
- Все балансы в системе (506 810 650 сум) — админские начисления, не живые деньги.

**Решение:** довести Payme до сертификации (см. `docs/PAYME_QUESTIONS.md`) ИЛИ
запустить на Click как основном канале, протестировав реальный платёж на 1000 сум.
Без этого клиент не может пополнить баланс → не может создать заказ → 0 выручки.

### 3. GitHub Actions заблокированы (биллинг)

```
The job was not started because your account is locked due to a billing issue.
```

Последствия: не работает CI, автодеплой и **сборка APK**. Каждый деплой — вручную
через `railway up`. Новые версии приложения не собираются.

**Решение:** оплатить GitHub. Это ~10 минут и разблокирует три процесса сразу.

## 🟠 ВАЖНОЕ — работает, но с риском

### 4. IP-whitelist Payme не активен в проде

`PAYME_WEBHOOK_WHITELIST` не задана → middleware пропускает запросы с любого IP.
Защита есть только по merchant-ключу.

**Решение:** прописать в Railway список IP Payme (уже есть в `.env.example`).

### 5. ИКПУ для фискализации не задан в проде

`PAYME_FISCAL_IKPU` не задана → чеки уйдут без кода 11201001001000000,
который вы согласовали.

**Решение:** прописать `PAYME_FISCAL_IKPU`, `PAYME_FISCAL_VAT_PERCENT` в Railway.

### 6. AI принимает мусор как валидный заказ (50%)

Замерено на реальных данных: чёрное фото + бессмысленный текст → смета на 300 000 сум
с уверенностью 90. Клиенту блокируются деньги в эскроу, мастер едет на несуществующую работу.

**Решение:** правило отказа в промпте + порог confidence. Делать после того, как
наберётся eval-набор (иначе непонятно, стало лучше или хуже).

### 7. Домен masteruz.uz не резолвится

DNS не настроен. Всё работает на `masteruz-frontend-production.up.railway.app`.
Для пользователей это выглядит несолидно и мешает доверию в платёжной воронке.

**Решение:** настроить DNS на Railway либо убрать домен из материалов.

### 8. Нет карт (Яндекс.Карты)

`VITE_YANDEX_MAPS_KEY=__SET_ME__`. Платформа геозависимая: подбор мастеров по
расстоянию, страница карты, адреса. Без ключа этот пласт функциональности мёртв.

**Решение:** получить ключ Яндекс.Карт и прописать во фронтенд-переменные.

## 🟡 СТОИТ СДЕЛАТЬ

### 9. Нет аналитики и мониторинга фронтенда
`VITE_SENTRY_DSN` и `VITE_YM_COUNTER` — заглушки. Ошибки у пользователей не видны,
поведение не измеряется. Бэкенд-Sentry настроен, фронтенд — нет.

### 10. Локальный реестр хранится в файлах
`local-registry.service.ts` пишет в `data/*.json` на диске. На Railway это
переживает рестарт только при подключённом volume. В коде честный TODO —
перенести в Postgres.

### 11. Тесты не покрывают ключевые модули
Есть 240 тестов, но нет ни одного на: instant-order (основной поток AI-заказа),
balance/escrow (движение денег), chat, estimation. Покрыты платежи, заказы,
i18n, безопасность, изменение цены.
