## 🔥 КРИТИЧНО — угроза данным и деньгам

### [ ] 1. Бэкапы БД не делаются больше месяца

Последний успешный бэкап — **15 июня 2026**. С тех пор 33 провала подряд
(workflow падает за 3-5 секунд — это блокировка биллинга GitHub).

В базе: 254 пользователя, балансы на 506 млн сум, заказы, переписка.
**Потеря БД сейчас = потеря всего бизнеса без возможности восстановления.**

- [ ] Снять ручной дамп сегодня же (`pg_dump` через публичный proxy-URL)
- [ ] Проверить, включены ли автобэкапы на стороне Railway
- [ ] После оплаты GitHub — убедиться, что workflow снова зелёный
- [ ] Настроить алерт на провал бэкапа (сейчас он падает молча)

### [ ] 2. Мастер не может вывести заработанные деньги

Механизма вывода **нет вообще**: нет модели заявки на вывод, нет эндпоинта,
нет экрана. Есть только тип транзакции `PAYOUT`, который начисляет сумму
на внутренний баланс мастера.

То есть: мастер выполнил работу → деньги упали на внутренний счёт → **застряли навсегда**.

Первый же мастер, закрывший заказ, обнаружит это и уйдёт. Это не «фича на потом» —
без вывода средств маркетплейс не работает как бизнес.

- [ ] Спроектировать вывод: модель `WithdrawalRequest` (сумма, карта, статус, комиссия)
- [ ] Экран мастера «Вывести средства» + история выводов
- [ ] Админский экран обработки заявок
- [ ] Решить канал выплаты: ручные переводы на первом этапе или API банка/Payme
- [ ] Минимальная сумма вывода, комиссия, сроки — прописать в оферте

### [ ] 3. Нет способа принять реальные деньги

- Payme: сертификация не пройдена, боевые креды не подтверждены.
  `PAYME_USE_SANDBOX` в проде не задана → код считает себя в боевом режиме.
- Click: ключи есть, webhook ни разу не проверен живым платежом.
- Все 506 млн сум на балансах — админские начисления, не живые деньги.

- [ ] Выбрать канал запуска (Click быстрее, Payme дольше но с холдированием)
- [ ] Провести один реальный платёж на 1000 сум end-to-end
- [ ] Явно задать `PAYME_USE_SANDBOX` в Railway
- [ ] Проверить возврат средств (refund) на реальной операции

## 🔴 БЛОКЕРЫ ЗАПУСКА

### [ ] 4. Нет мастеров: 247 клиентов на 4 мастера

Из 6 профилей только **3 имеют категории**. Рассылка новых заказов идёт по
категориям → заказ физически видят 3 человека. Откликов за всё время — 2.

- [ ] Ручной набор 30–50 мастеров по ключевым категориям
- [ ] Сделать выбор категорий обязательным при регистрации мастера
- [ ] Решить судьбу регистрационного взноса 400 000 сум на период набора
      (сейчас он не блокирует работу технически, но отпугивает в онбординге)
- [ ] Продумать, что мастер видит в первые 5 минут после регистрации

### [ ] 5. GitHub Actions заблокированы по биллингу

```
The job was not started because your account is locked due to a billing issue.
```

Ломает сразу четыре процесса: CI, автодеплой, сборку APK и **бэкапы БД**.

- [ ] Оплатить GitHub (~10 минут, разблокирует всё перечисленное)

### [ ] 6. Ноль завершённых заказов — сквозной путь не проверен

8 заказов, 0 доведённых до конца. Значит **ни разу не проверены в бою**:
эскроу, расчёт комиссии, выплата мастеру, отзывы, гарантии, закрытие сделки.

- [ ] Провести 3–5 сквозных заказов вручную до статуса COMPLETED
- [ ] Проверить, что деньги списались/начислились корректно
- [ ] Проверить уведомления на каждом шаге

## 🟠 РИСКИ В ПРОДЕ

### [ ] 7. IP-whitelist Payme не активен
`PAYME_WEBHOOK_WHITELIST` не задана → middleware пропускает любой IP.
- [ ] Прописать список IP Payme в Railway (уже есть в `.env.example`)

### [ ] 8. ИКПУ для фискализации не задан
`PAYME_FISCAL_IKPU` не задана → чеки уйдут без согласованного кода.
- [ ] Прописать `PAYME_FISCAL_IKPU=11201001001000000` и `PAYME_FISCAL_VAT_PERCENT`

### [ ] 9. AI принимает мусорные заявки (50%)
Замерено: чёрное фото + бессмысленный текст → смета на 300 000 сум, confidence 90.
Клиенту блокируются деньги, мастер едет на несуществующую работу.
- [ ] Довести eval-набор до 30+ кейсов
- [ ] Правило отказа в промпте + порог confidence
- [ ] Проверить прогоном, что стало лучше

### [ ] 10. Домен masteruz.uz не резолвится
Всё работает на `masteruz-frontend-production.up.railway.app`.
В платёжной воронке это бьёт по доверию.
- [ ] Настроить DNS либо убрать домен из всех материалов и оферты

### [ ] 11. Нет карт (Яндекс.Карты)
`VITE_YANDEX_MAPS_KEY=__SET_ME__`. Платформа геозависимая: подбор по расстоянию,
страница карты, адреса.
- [ ] Получить ключ и прописать во фронтенд

## 🟡 КАЧЕСТВО И НАБЛЮДАЕМОСТЬ

### [ ] 12. Фронтенд без мониторинга и аналитики
`VITE_SENTRY_DSN` и `VITE_YM_COUNTER` — заглушки. Ошибки у пользователей не видны,
поведение не измеряется. На бэкенде Sentry настроен.
- [ ] Подключить Sentry фронтенда
- [ ] Подключить аналитику (Яндекс.Метрика) — без неё непонятно, где отваливается воронка

### [ ] 13. Тесты не покрывают ключевые модули
240 тестов, но нет ни одного на: instant-order (основной AI-поток), balance/escrow
(движение денег), chat, estimation.
- [ ] Тесты на баланс и эскроу — там живут деньги
- [ ] Тесты на instant-order

### [ ] 14. local-registry хранится в файлах
`data/*.json` на диске. На Railway переживёт рестарт только с подключённым volume.
В коде честный TODO.
- [ ] Перенести в Postgres либо подтвердить volume

### [ ] 15. iOS-сборка не выпущена
Проект `frontend/ios` существует, но приложения нет. Промо в Mini App показывается
только Android-пользователям.
- [ ] Решить: выпускать iOS или официально заявить Android-only
